# 人事給与システム別リポジトリ化要件（TA連携）

## 1. 目的
- 就業管理システム TA とは**別リポジトリ**で、人事給与システムをパッケージとして構築する。
- 勤怠システムと人事給与システムは疎結合で連携し、将来的な別販路・別導入・別リリースを可能にする。
- モバイル対応はレスポンシブ止まりではなく、**iOS / Android のネイティブアプリ**も前提に設計する。

## 2. システム境界（初期方針）
### 2-1. 現在の勤怠システム（本リポジトリ）が担う範囲
- 打刻（出退勤）
- シフト管理
- 休暇申請/管理
- 勤怠実績の保持
- 人事給与システムへ連携するための勤怠元データの提供

### 2-2. 新規の人事給与システム（別リポジトリ）が担う範囲
- 従業員マスタ管理
- 部門・役職・雇用区分管理
- 入社・退職・異動などの人事イベント管理
- 締め処理
- 月次集計
- 給与システム向け出力
- 人事給与向け承認、監査、設定管理

### 2-3. 連携方針
- 連携は **API ファースト** を基本とする。
- リリース、デプロイ、障害対応、バージョン管理はリポジトリ単位で独立させる。
- 勤怠データは本システムから人事給与システムへ連携する。
- 従業員・組織情報の正本は人事給与システム側で持つ想定を基本案とする。

## 3. 対象範囲（MVP）
- 人事給与システム用の Web 管理画面
- iOS / Android 向けネイティブアプリ
- 従業員マスタ管理
- 部門・役職・雇用区分管理
- 入社・退職・異動の履歴管理
- 勤怠データの締め処理
- 法定/所定の残業・深夜・休日の集計
- 給与システム向け CSV / 連携データ出力
- 申請/承認ステータスを反映した確定データ作成
- 勤怠システムとのデータ連携仕様整理

## 4. 非対象（このフェーズでは実施しない）
- 社会保険/税計算そのもの
- 人事評価機能
- 採用管理
- 会計システムとの包括連携
- モバイルアプリのオフライン完全対応

## 5. 機能要件
### 5-1. 人事管理
- 従業員の基本情報、所属部門、役職、雇用区分を管理可能
- 入社日、退職日、異動履歴を保持可能
- 有効/無効、在籍状態を管理可能

### 5-2. 組織管理
- 部門の階層管理が可能
- 管理者、マネージャー、一般従業員などの権限区分を扱える

### 5-3. 締め設定
- 月次締め日、起算日、締め時刻を設定可能
- 再締め時の差分検出（再計算対象の明示）

### 5-4. 集計エンジン
- 日次明細から月次集計を生成
- 丸め設定（15分/30分/60分、切上げ/切捨て/四捨五入）
- 深夜帯（22:00-05:00）区分

### 5-5. 出力・外部連携
- 出力項目マッピングを設定で変更可能
- エンコーディング（UTF-8/Shift_JIS）切替
- ヘッダー有無、区切り文字を設定可能
- 勤怠システムから受領したデータを締め対象へ反映できる
- API 連携失敗時の再送・再実行状況を確認できる

### 5-6. 承認連携
- 未承認レコードは締め対象外にできる
- 承認済みのみで確定ファイルを作成可能

### 5-7. モバイルアプリ
- iOS / Android 向けにネイティブアプリを提供する
- 従業員向けに、人事情報参照、申請状況確認、給与関連通知を提供できる
- 管理者向けに、承認・差戻し・締め結果確認の主要導線を提供できる
- 将来的なプッシュ通知連携を想定した設計にする

### 5-8. 監査ログ
- 従業員情報変更の履歴を記録
- 締め実行者、実行時刻、対象期間、件数、再実行有無を記録
- 連携 API の実行結果を記録

## 6. 連携要件
- 勤怠システムと人事給与システムの責務境界を文書化する
- 連携 API または連携バッチの入出力仕様を定義する
- 連携失敗時の再実行手順を定義する
- 将来的に SSO または共通認証基盤へ寄せられる構成を意識する

## 7. 技術要件（初期案）
- 人事給与システムは別リポジトリで管理する
- Web とモバイルで共通利用できる API / ドメイン設計を行う
- モバイルアプリは iOS / Android のネイティブ実装を前提にする
- 設定駆動で会社差分を吸収し、コード分岐を最小化する

## 8. データ要件
- 新規テーブル候補:
  - employees
  - departments
  - employee_positions
  - employee_status_history
  - organization_settings
  - payroll_periods
  - payroll_results
  - payroll_export_jobs
  - payroll_settings
  - integration_jobs
  - audit_logs

## 9. 受け入れ条件
- 人事給与システムが本システムとは**別リポジトリ前提**で整理されている
- 勤怠システムとの連携境界が定義されている
- 従業員の所属・役職・雇用区分変更を履歴付きで管理できる
- 入社・退職・異動が月次集計対象に正しく反映される
- 指定締め日の月次集計が UI と CSV で一致する
- 会社 A/B で異なる CSV フォーマットを設定だけで切替できる
- 未承認データ除外 ON 時に、対象外件数が確認できる
- 同一期間再実行時に差分件数を表示できる
- iOS / Android ネイティブアプリ向けの対象機能と責務が定義されている

## 10. 実装順序（推奨）
1. システム境界と連携責務を定義する
2. 別リポジトリ用のアーキテクチャ案を確定する
3. 人事データモデルを追加する（従業員属性、役職、異動履歴）
4. 給与データモデルを追加する（締め期間、集計結果、出力履歴、設定）
5. 連携 API / バッチ仕様を定義する
6. 集計ロジックをサービス層へ分離する
7. 締め処理ユースケースを実装する
8. CSV エクスポート設定機構を実装する
9. モバイルアプリ要件を Web/API 設計へ反映する
10. 監査ログと運用手順を整備する

## 11. リスクと対策
- リスク: 会社ごとの差分が増えて分岐地獄になる
- 対策: 設定駆動 + 変換アダプタ層を採用する

- リスク: 勤怠システムと人事給与システムの責務が曖昧になる
- 対策: データ正本、更新責任、連携方向を先に文書化する

- リスク: モバイル向け要件を後付けすると API が破綻する
- 対策: 初期段階から API ファーストで設計し、モバイル導線を受け入れ条件へ含める

- リスク: 認証/権限未整備で本番運用できない
- 対策: 本番前に共通認証方針とロール認可を必須タスク化する
